Структурированный промпт для ИИ-агента на настройку сбора метрик, их агрегации и отображения статистики с heatmap в админ-панели:

Задача: Реализовать сбор поведенческих метрик на клиентской странице, их отправку на бэкенд раз в секунду, хранение без лишних проверок, и отображение агрегированной статистики (максимальное время, heatmap курсора) в админ-панели.

Часть 1. Клиентский скрипт сбора метрик (behavior-metrics.js)
1.1. Создание файла
Файл: frontend-client/src/tracking/behavior-metrics.js
Скрипт подключается на основную клиентскую страницу (/ — страницу заявки).

1.2. Собираемые метрики
Метрика	Поле	Тип	Описание
Время на странице	time_on_page	integer (секунды)	Счётчик с момента загрузки, увеличивается каждую секунду
Нажатые кнопки	buttons_clicked	string (JSON)	Объект: { "button_id_or_text": count, ... }
Позиции курсора	cursor_positions	string (JSON)	Массив координат за последнюю секунду: [{ "x": 450, "y": 320, "timestamp": ... }, ...]
Частота возвратов	return_frequency	integer	Количество предыдущих визитов (из sessionStorage / localStorage)

1.3. Логика сбора
Инициализация при загрузке:
1. Засечь время загрузки страницы (performance.timing или Date.now)
2. Инициализировать объект метрик:
   - time_on_page = 0
   - buttons_clicked = {}  // словарь: ключ — идентификатор кнопки, значение — счётчик
   - cursor_positions = [] // массив координат за текущую секунду
   - return_frequency = получить из localStorage (если первый визит — 0, иначе инкрементировать и сохранить)

3. Повесить слушатели:
   - mousemove: сохранять координаты { x: event.clientX, y: event.clientY } в cursor_positions
   - click на кнопках/элементах: buttons_clicked[element_identifier]++

4. Запустить setInterval с периодом 1000ms → функция отправки метрик

Функция отправки (каждую секунду):
1. Обновить time_on_page: += 1 (прошла 1 секунда)
2. Сериализовать текущий массив cursor_positions в JSON-строку
3. Сериализовать buttons_clicked в JSON-строку
4. Отправить POST /api/behavior-metrics/ с телом:
   {
     "application_id": 0,       // жёстко задано, бэкенд игнорирует
     "time_on_page": <число>,
     "buttons_clicked": "<JSON-строка>",
     "cursor_positions": "<JSON-строка>",
     "return_frequency": <число>
   }
5. Очистить cursor_positions = [] (для следующей секунды)
6. Не очищать buttons_clicked (накапливаем за всю сессию)
Завершение сессии (beforeunload / pagehide):

Отправить финальный пакет метрик перед закрытием страницы (использовать navigator.sendBeacon для надёжности)

1.4. Идентификация элементов для buttons_clicked
При клике на интерактивные элементы определять их идентификатор:

Приоритет: data-metric-id → id → name → текст кнопки (обрезка до 30 символов)

Рекомендуется проставить атрибут data-metric-id на ключевые элементы клиентской страницы (кнопка отправки формы, ползунки, селекты)

1.5. Подключение скрипта
На клиентскую страницу (index.html или главный компонент) добавить:
<script src="/src/tracking/behavior-metrics.js" type="module"></script>

Скрипт должен загружаться и запускаться автоматически при открытии страницы.

Часть 2. Бэкенд: эндпоинт /api/behavior-metrics
2.1. Модель BehaviorMetric (новая или существующая)
Таблица: behavior_metrics
Поле	Тип	Ограничения	Описание
id	SERIAL / INTEGER	PRIMARY KEY	Автоинкремент
application_id	INTEGER	DEFAULT 0	Всегда 0, игнорировать при вставке
time_on_page	INTEGER	DEFAULT 0	Накопленные секунды на момент отправки
buttons_clicked	TEXT / JSONB	NULLABLE	JSON-строка счётчиков кнопок
cursor_positions	TEXT / JSONB	NULLABLE	JSON-строка массива координат
return_frequency	INTEGER	DEFAULT 0	Частота возвратов
created_at	TIMESTAMP	DEFAULT NOW()	Время записи метрики

Ключевые требования:
Никаких проверок application_id! Значение просто игнорируется (принимается, сохраняется как есть, но не валидируется на существование в других таблицах).
Не проверять существование записи для application_id = 0.
Метрики всегда создаются новой записью (INSERT, не UPSERT, не UPDATE).
Все поля опциональные при приёме (Pydantic-схема с Optional).

2.2. Pydantic-схема
Файл: backend/schemas/behavior.py
python
from pydantic import BaseModel, Field
from typing import Optional
from datetime import datetime

class BehaviorMetricCreate(BaseModel):
    application_id: Optional[int] = 0
    time_on_page: Optional[int] = 0
    buttons_clicked: Optional[str] = None
    cursor_positions: Optional[str] = None
    return_frequency: Optional[int] = 0

class BehaviorMetricResponse(BaseModel):
    id: int
    application_id: int
    time_on_page: int
    buttons_clicked: Optional[str]
    cursor_positions: Optional[str]
    return_frequency: int
    created_at: datetime

    class Config:
        from_attributes = True

2.3. CRUD и роуты
Файл: backend/routes/behavior.py
Метод	Маршрут	Описание	Защита
POST	/api/behavior-metrics/	Создать запись метрики	❌ Без авторизации
GET	/api/behavior-metrics/	Получить список метрик (с пагинацией)	✅ Только админ
GET	/api/behavior-metrics/stats	Получить агрегированную статистику (см. Часть 3)	✅ Только админ

POST /api/behavior-metrics/:
Принимает BehaviorMetricCreate
Просто создаёт новую запись в таблице, без проверок application_id
Возвращает 201 Created + созданный объект

GET /api/behavior-metrics/:
Параметры запроса: skip (int, default 0), limit (int, default 100)
Возвращает массив метрик, отсортированных по created_at DESC
Требует авторизацию админа (Depends(get_current_admin))

Важное исправление: Убедиться, что в бэкенде нет логики проверки существования application_id в какой-либо таблице applications. Если такая проверка есть — удалить её полностью. Метрики должны быть анонимными, application_id просто сохраняется как переданное значение без валидации.

Часть 3. Агрегация статистики

3.1. Эндпоинт GET /api/behavior-metrics/stats
Файл: backend/routes/behavior.py
Ответ:
json
{
  "avg_time_on_page": {
    "daily": 45,      // среднее макс. время в секундах за сегодня
    "weekly": 320,    // среднее макс. время в секундах за 7 дней
    "monthly": 1200   // среднее макс. время в секундах за 30 дней
  },
  "max_time_on_page": {
    "daily": 120,
    "weekly": 850,
    "monthly": 3400
  },
  "top_buttons": [
    { "button": "submit-form", "clicks": 45 },
    { "button": "budget-slider", "clicks": 32 }
  ],
  "heatmap_data": [
    { "x": 450, "y": 320, "frequency": 120 },
    { "x": 600, "y": 200, "frequency": 95 }
  ],
  "total_sessions": 256,
  "total_metrics_records": 12500
}

Логика расчёта на бэкенде:
1. Время (daily/weekly/monthly):
Сгруппировать метрики по дате (DATE(created_at))
Для каждой сессии (условно — группа близких по времени записей с одного IP или просто максимальное time_on_page в каждой группе) взять максимальное значение time_on_page
Усреднить по периодам

2. Топ кнопок:
Распарсить все buttons_clicked (JSON), агрегировать сумму кликов по каждому идентификатору кнопки
Вернуть топ-10

3. Heatmap данные:
Распарсить все cursor_positions (JSON), извлечь все координаты
Округлить x и y до ближайших ~20px (группировка по ячейкам сетки)
Подсчитать частоту попаданий в каждую ячейку
Вернуть массив { x, y, frequency }, где frequency — количество попаданий

Оптимизация: если метрик очень много, можно:
Ограничить выборку периодом (daily — 1 день, weekly — 7 дней, monthly — 30 дней)
Для heatmap — ограничить выборку последними N записями или использовать агрегацию на стороне БД

Часть 5. Исправление ошибок и важные указания

5.1. Ошибка: «Behavior metrics for application 0 already exists»
Проблема: Бэкенд проверяет существование записи для application_id = 0 (возможно, через внешний ключ на несуществующую таблицу applications).

Решение:
Убрать любые проверки существования application_id в других таблицах.
Убрать внешний ключ, если он есть.
Убрать уникальные ограничения на application_id.
Метрики — только INSERT новых записей, без поиска существующих.
application_id — просто целочисленное поле, которое принимается от фронтенда и сохраняется без валидации.

5.2. SQL-команды для исправления
Если ИИ-агент обнаружит проблемы с таблицей — выполнить следующие SQL (после проверки):
sql
-- Удалить внешний ключ, если есть
ALTER TABLE behavior_metrics DROP CONSTRAINT IF EXISTS fk_behavior_metrics_application;
-- Удалить уникальный индекс, если есть
DROP INDEX IF EXISTS idx_behavior_metrics_app_id;
-- Убедиться, что поля опциональные
ALTER TABLE behavior_metrics ALTER COLUMN application_id DROP NOT NULL;
ALTER TABLE behavior_metrics ALTER COLUMN buttons_clicked DROP NOT NULL;
ALTER TABLE behavior_metrics ALTER COLUMN cursor_positions DROP NOT NULL;
ALTER TABLE behavior_metrics ALTER COLUMN return_frequency SET DEFAULT 0;

5.3. Проверка формата данных
Убедиться, что buttons_clicked и cursor_positions — это текстовые поля (TEXT или JSONB), которые принимают любые строки.
Фронтенд всегда отправляет JSON-строки для этих полей.
Бэкенд не парсит их при вставке (просто сохраняет как строку), парсинг — только при агрегации.
