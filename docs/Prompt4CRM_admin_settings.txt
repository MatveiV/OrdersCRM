Промпт для ИИ-агента на разработку CRUD-интерфейса управления услугами в админ-панели:

Задача: Реализовать в админ-панели (admin.html / /admin/dashboard) визуальный интерфейс для управления услугами (таблица admin_settings) с полным CRUD-функционалом, профессиональным спокойным дизайном и валидацией данных. По завершении — сформировать Markdown-документацию API-эндпоинтов для фронтенда.

1. Бэкенд: доработка и проверка эндпоинтов
1.1. Модель AdminSetting (существующая, проверить/дополнить)
Таблица: admin_settings
Поля:
id — PK, автоинкремент
service_name — название услуги (строка, NOT NULL)
budget_range — диапазон бюджета (строка/JSON, например {"min": 50000, "max": 500000, "step": 10000})
task_type — тип задачи (строка, опционально)
product_interest — интересующий продукт (строка, опционально)
description — описание услуги (текст, опционально)
is_active — активна ли услуга (boolean, DEFAULT TRUE)
created_at — дата создания (TIMESTAMP, DEFAULT NOW())
updated_at — дата обновления (TIMESTAMP, DEFAULT NOW(), автообновление)

1.2. CRUD-операции в модели
В классе AdminSettingCRUD должны быть методы:
get_all() — получить все услуги
get_by_id(id) — получить услугу по ID
create(data: dict) — создать новую услугу
update(id, data: dict) — обновить услугу
delete(id) — удалить услугу (или soft-delete: is_active = False)

1.3. Роуты (проверить и дополнить)
Файл: backend/routes/admin.py
Все эндпоинты защищены Depends(get_current_admin).
Метод	Маршрут	Входные данные	Ответ
GET	/admin/services	—	Массив объектов услуг [{...}, {...}]
GET	/admin/services/{id}	—	Объект услуги {...} или 404
POST	/admin/services	JSON-объект услуги	Созданный объект {...} + статус 201
PUT	/admin/services/{id}	JSON-объект с обновлёнными полями	Обновлённый объект {...}
DELETE	/admin/services/{id}	—	{ "success": true } или 404

Требования к обработке:
Валидация входных данных: использовать Pydantic-схемы для каждого эндпоинта
POST и PUT: проверять, что service_name не пустой, обязательные поля присутствуют
Ответ GET /admin/services всегда возвращает массив (даже если услуг нет — возвращать пустой массив [], а не null или объект)
Ошибки валидации: возвращать 422 Unprocessable Entity с понятным описанием ошибки
Отсутствие записи: возвращать 404 Not Found

1.4. Pydantic-схемы
Файл: backend/schemas/admin.py (создать, если нет)
python
from pydantic import BaseModel, Field
from typing import Optional
from datetime import datetime

class AdminSettingBase(BaseModel):
    service_name: str = Field(..., min_length=1, max_length=200)
    budget_range: Optional[str] = None
    task_type: Optional[str] = None
    product_interest: Optional[str] = None
    description: Optional[str] = None
    is_active: bool = True

class AdminSettingCreate(AdminSettingBase):
    pass

class AdminSettingUpdate(BaseModel):
    service_name: Optional[str] = Field(None, min_length=1, max_length=200)
    budget_range: Optional[str] = None
    task_type: Optional[str] = None
    product_interest: Optional[str] = None
    description: Optional[str] = None
    is_active: Optional[bool] = None

class AdminSettingResponse(AdminSettingBase):
    id: int
    created_at: datetime
    updated_at: datetime

    class Config:
        from_attributes = True

2.4. Кнопка «Добавить услугу» и модальное окно
По нажатию на кнопку «+ Добавить услугу» открывается модальное окно:

Поля формы:
Поле	Тип	Обязательное	Описание
Название услуги	input text	✅	service_name
Тип задачи	input text	❌	task_type
Интересующий продукт	input text	❌	product_interest
Диапазон бюджета (от)	input number	❌	Минимальная сумма
Диапазон бюджета (до)	input number	❌	Максимальная сумма
Шаг ползунка	input number	❌	Шаг для range slider на клиентской форме
Описание	textarea	❌	description
Активна	checkbox	—	is_active (по умолчанию включён)

Логика:
При отправке — POST /admin/services
Поля «от», «до», «шаг» объединяются в budget_range как JSON-строка: {"min": ..., "max": ..., "step": ...}
Валидация на фронте: название услуги не пустое, максимальная сумма > минимальной
При успехе — модальное окно закрывается, таблица обновляется (повторный GET или добавление строки из ответа)

2.5. Редактирование услуги
По нажатию на иконку ✏️ открывается то же модальное окно, но:
Поля предзаполнены текущими значениями
При разборе budget_range (JSON) — значения извлекаются в поля «от», «до», «шаг»
При отправке — PUT /admin/services/{id}
Кнопка сохранения: «Сохранить изменения»

2.6. Удаление услуги
По нажатию на иконку 🗑️:
Показать диалог подтверждения: «Вы уверены, что хотите удалить услугу "{service_name}"? Это действие нельзя отменить.»
При подтверждении — DELETE /admin/services/{id}
При успехе — строка удаляется из таблицы
Если ошибка — показать уведомление

2.7. Уведомления (Toast-сообщения)
Реализовать систему toast-уведомлений:
✅ Успех: «Услуга успешно добавлена» / «Изменения сохранены» / «Услуга удалена»
❌ Ошибка: «Не удалось загрузить список услуг» / «Ошибка при сохранении» / «Услуга не найдена»
Уведомления автоматически исчезают через 3-5 секунд
Цвета: зелёный для успеха, красный для ошибки, серый для информации

3. Обработка ошибок и валидация
3.1. Фронтенд
Все запросы к API обёрнуты в try/catch
При ответе с ошибкой — показывать сообщение пользователю (не «сырой» JSON)
Перед отправкой формы — проверка обязательных полей
GET /admin/services → проверить, что ответ — массив (Array.isArray). Если нет — показать ошибку «Некорректный формат данных от сервера»

3.2. Бэкенд
Все входящие данные валидируются через Pydantic-схемы
GET /admin/services гарантированно возвращает массив (даже пустой)
POST / PUT → при невалидных данных возвращать 422 с полями ошибок
GET / PUT / DELETE по несуществующему ID → 404

## GET /admin/services
Получить список всех услуг.
### Ответ (200 OK)
```json
[
  {
    "id": 1,
    "service_name": "Консультация",
    "budget_range": "{\"min\": 50000, \"max\": 500000, \"step\": 10000}",
    "task_type": "Аналитика",
    "product_interest": "AI-агент",
    "description": "Первичная консультация по проекту",
    "is_active": true,
    "created_at": "2025-01-15T10:30:00Z",
    "updated_at": "2025-01-20T14:00:00Z"
  }
]
Важно: Ответ всегда массив. Если услуг нет — [].

GET /admin/services/{id}
Получить услугу по ID.
Ответ (200 OK)
Один объект услуги (см. выше).
Ошибки
404 Not Found — услуга не найдена

POST /admin/services
Создать новую услугу.
Тело запроса
json
{
  "service_name": "Разработка MVP",
  "budget_range": "{\"min\": 100000, \"max\": 1000000, \"step\": 50000}",
  "task_type": "Разработка",
  "product_interest": "Торговая платформа",
  "description": "Разработка MVP под ключ",
  "is_active": true
}
Ответ (201 Created)
Созданный объект услуги с id.
Ошибки: 422 Unprocessable Entity — ошибка валидации (service_name пустой и т.д.)

PUT /admin/services/{id}
Обновить услугу.
Тело запроса
json
{
  "service_name": "Новое название",
  "is_active": false
}
Можно передать только изменяемые поля.
Ответ (200 OK)
Обновлённый объект услуги.
Ошибки:
404 Not Found
422 Unprocessable Entity

DELETE /admin/services/{id}
Удалить услугу.
Ответ (200 OK)
json
{
  "success": true,
  "message": "Услуга удалена"
}
Ошибки
404 Not Found
