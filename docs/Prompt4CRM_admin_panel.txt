Промпт для разработки системы авторизации админ-панели в Orders CRM:

Задача: Реализовать систему авторизации для админ-панели CRM с JWT-токенами, регистрацией первого администратора и защитой маршрутов.

1. Backend (FastAPI)
1.1. Новая модель: AdminUser (администратор)
Назначение: хранение учётных данных администраторов с хешированными паролями.

Поля таблицы admin_users:
Поле	Тип	Ограничения	Описание
id	SERIAL / INTEGER	PRIMARY KEY, AUTOINCREMENT	Уникальный идентификатор
username	VARCHAR(100)	UNIQUE, NOT NULL	Логин администратора
password_hash	VARCHAR(255)	NOT NULL	Хеш пароля (bcrypt)
created_at	TIMESTAMP	DEFAULT NOW()	Дата создания учётной записи
is_active	BOOLEAN	DEFAULT TRUE	Активен ли администратор

Требования к реализации:
В классе модели докстрингом прописать SQL-код для генерации таблицы (см. раздел 5).
Сразу под классом модели создать класс AdminUserCRUD со следующими методами:
create_admin(username: str, password: str) — создать администратора (пароль хешировать через bcrypt)
get_admin_by_id(admin_id: int) — получить по ID
get_admin_by_username(username: str) — получить по логину
get_all_admins() — получить список всех администраторов
update_admin(admin_id: int, **kwargs) — обновить данные
delete_admin(admin_id: int) — удалить администратора
get_admin_count() — получить количество администраторов (нужно фронтенду для определения, показывать ли кнопку регистрации)
verify_password(username: str, password: str) — проверить пароль (вернуть AdminUser или None)

Файл: backend/models/admin_user.py

1.2. Роуты аутентификации
Файл: backend/routes/auth.py

Реализовать следующие эндпоинты:
Метод	Маршрут	Описание
POST	/auth/register	Регистрация администратора (только если в БД нет ни одного админа)
POST	/auth/login	Вход: принимает username + password, возвращает JWT-токены
POST	/auth/refresh	Обновление access-токена по refresh-токену
GET	/auth/check	Проверить, есть ли администраторы в БД (возвращает { has_admins: true/false })
GET	/auth/me	Получить данные текущего администратора (защищённый маршрут)

Логика эндпоинтов:

POST /auth/register
Тело запроса: { "username": "string", "password": "string" }
Проверка: если get_admin_count() > 0 → ошибка 403 Forbidden («Регистрация закрыта, администратор уже существует»)
Действие: создать админа через AdminUserCRUD.create_admin()
Ответ: 201 Created + JWT-токены (как при логине)

POST /auth/login
Тело запроса: { "username": "string", "password": "string" }
Проверка: AdminUserCRUD.verify_password(username, password)
Если неверно: 401 Unauthorized («Неверный логин или пароль»)
Если верно: сгенерировать пару токенов:
access_token — JWT, срок жизни 15-30 минут, payload: { sub: admin.id, username: admin.username, type: "access" }
refresh_token — JWT, срок жизни 7 дней, payload: { sub: admin.id, type: "refresh" }
Ответ: 200 OK + { access_token, refresh_token, token_type: "bearer" }

POST /auth/refresh
Тело запроса: { "refresh_token": "string" }
Проверка: валидность refresh-токена, тип "refresh"
Ответ: новый access_token и refresh_token

GET /auth/check
Ответ: { "has_admins": true/false, "count": N }
Используется фронтендом для принятия решения: показывать форму входа или регистрации.

GET /auth/me (защищённый)
Заголовок: Authorization: Bearer <access_token>
Ответ: { id, username, created_at, is_active } — данные текущего админа

1.3. JWT и безопасность
Файл: backend/core/security.py

Секретный ключ для JWT — брать из переменных окружения (JWT_SECRET_KEY), в docker-compose / .env
Алгоритм: HS256
Хеширование паролей: bcrypt (через passlib или прямой вызов bcrypt)
Функция-зависимость get_current_admin — извлекает и валидирует access-токен из заголовка Authorization, возвращает объект админа. Использовать как Depends(get_current_admin) в защищённых роутах.

1.4. Защита существующих админских роутов
Все существующие роуты админ-панели (управление AdminData, просмотр лидов и т.д.) должны быть защищены зависимостью Depends(get_current_admin). Без валидного токена — 401 Unauthorized.

1.5. Подключение роутов в main.py
Добавить auth_router из routes/auth.py

Убедиться, что CORS настроен корректно (фронтенд сможет слать запросы с авторизационными заголовками)

2. Frontend (Админ-панель)
2.1. Страница /admin

При переходе на /admin фронтенд выполняет логику:

Шаг 0. Проверка токена в localStorage:
Если access_token есть → попытка запроса GET /auth/me
Успешно → редирект на /admin/dashboard (сама админ-панель)
Ошибка 401 → попытка обновить токен через POST /auth/refresh
Успешно → сохранить новые токены, редирект на /admin/dashboard
Ошибка → очистить токены, показать страницу входа
Если access_token отсутствует → показать страницу входа.

Шаг 1. Загрузка страницы входа:
Сразу сделать запрос GET /auth/check
Если has_admins: false → показать форму регистрации (поля: логин, пароль, подтверждение пароля; кнопка «Зарегистрироваться»)
Если has_admins: true → показать форму входа (поля: логин, пароль; кнопка «Войти»). Кнопка «Зарегистрироваться» отсутствует.

2.2. Форма регистрации (только если админов нет)
Поля: username, password, confirm_password
Валидация: пароли совпадают, минимальная длина 6 символов
Отправка: POST /auth/register
При успехе: сохранить полученные токены в localStorage, редирект на /admin/dashboard

2.3. Форма входа
Поля: username, password
Отправка: POST /auth/login
При успехе: сохранить токены в localStorage (access_token, refresh_token), редирект на /admin/dashboard
При ошибке: показать сообщение «Неверный логин или пароль» (в стилистике админки)

2.4. Хранение токенов и автоматическое обновление
Токены хранить в localStorage

Перед каждым запросом к защищённым эндпоинтам проверять срок access_token (декодировать JWT на фронте)
Если access_token истекает → автоматически вызвать POST /auth/refresh, сохранить новые токены, повторить запрос
Если и refresh_token недействителен → редирект на страницу входа

2.5. Dashboard /admin/dashboard (заглушка)
После успешной авторизации — приветствие: «Добро пожаловать, {username}!»
Кнопка «Выйти» — очищает токены, редирект на /admin/login
Заглушка под будущий функционал админ-панели

2.6. Стилизация админ-панели
Тёмная тема (bg: #1a1a2e, #16213e)
Акценты — золотые (#D4AF37) или синие
Шрифт — округлый, без засечек (Nunito/Nunito Sans)
Минималистичный дизайн, форма по центру экрана

5. SQL-код для создания таблицы (выдать в конце работы)
sql
-- Таблица администраторов
CREATE TABLE IF NOT EXISTS admin_users (
    id SERIAL PRIMARY KEY,
    username VARCHAR(100) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    created_at TIMESTAMP DEFAULT NOW(),
    is_active BOOLEAN DEFAULT TRUE
);

-- Индекс для быстрого поиска по логину
CREATE INDEX IF NOT EXISTS idx_admin_users_username ON admin_users(username);

Итог: админ-панель получает полноценную JWT-авторизацию с регистрацией первого админа, защитой всех маршрутов, автообновлением токенов на фронтенде и стилизованным интерфейсом входа.
