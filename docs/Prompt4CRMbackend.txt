Задача: Разработать backend для CRM-системы на FastAPI.

Общие требования:
Приложение приватное, доступно только внутри машины (не выходить наружу). Проверить настройки прокидования портов в docker-compose.
Все данные хранятся и обрабатываются строго внутри контура.
Доступ к базе данных только через backend.

БД: PostgreSQL (параметры подключения взять из docker-compose).

Структура проекта (внутри папки backend):

backend/
├── main.py          — точка входа, объединение всех роутов
├── core/
│   └── database.py  — драйвер подключения к PostgreSQL
├── models/
│   ├── lead.py      — модель Lead + CRUD
│   ├── behavior.py  — модель Behavior + CRUD
│   └── admin.py     — модель AdminData + CRUD
└── routes/
    ├── lead.py      — FastAPI роуты для Lead
    ├── behavior.py  — FastAPI роуты для Behavior
    └── admin.py     — FastAPI роуты для AdminData
	
Модели данных

1. Модель Lead (основная заявка)
Поля:
id — PK, автоинкремент
first_name — имя (строка)
last_name — фамилия (строка)
middle_name — отчество (строка, опционально)
contact_data — контактные данные (строка, JSON или текст)
business_niche — ниша бизнеса (строка)
company_size — размер компании (строка)
task_volume — объем задачи (строка)
role — кем является: сотрудник или руководитель (строка)
business_info — информация о бизнесе (строка)
budget — бюджет (строка, на фронте выбирается ползунком)
project_deadline — срок, когда нужен результат (строка)
task_type — тип задачи (строка)
product_interest — интересующий продукт (строка)
preferred_contact_method — предпочтительный способ связи (строка)
convenient_time — удобное время связи (строка)
comment — комментарий (строка, опционально)

Требования:
В классе модели докстрингом прописать SQL-код для генерации таблицы.
Сразу под классом модели создать класс с CRUD-операциями (создание, чтение, обновление, удаление).

2. Модель Behavior (поведение пользователя на сайте)
Связь с Lead: один-к-одному по lead_id (PK, FK на Lead.id)

Поля:
lead_id — PK + FK на Lead
time_spent_seconds — время, проведенное на странице (int/float)
buttons_clicked — какие кнопки нажимал (JSON/текст)
cursor_hover_zones — где зависал курсором (JSON/текст)
return_count — как часто возвращался на страницу (int)
Другие агрегированные лид-метрики и техническая информация, собранные фронтендом.

Требования:
В классе модели докстрингом прописать SQL-код для генерации таблицы.
Сразу под классом модели создать класс с CRUD-операциями.

3. Модель AdminData (административные данные)
Назначение: хранит настройки, которые фронтенд использует для динамического формирования интерфейса.

Поля:
id — PK, автоинкремент
service_name — какие услуги предоставляем (строка)
budget_range — какой диапазон бюджета доступен для выбора (строка/JSON)
Другие поля по необходимости.

Требования:
В классе модели докстрингом прописать SQL-код для генерации таблицы.
Сразу под классом модели создать класс с CRUD-операциями.


Маршруты (Routes)
Для каждой модели создать отдельный файл роута в папке routes/.
В каждом роуте реализовать FastAPI-функции, которые вызывают соответствующие CRUD-операции из модуля моделей.
Пример: POST /leads, GET /leads/{id}, PUT /leads/{id}, DELETE /leads/{id}.

main.py
Объединить все роуты в общий FastAPI-роутер.
Подключить драйвер базы данных из core/database.py.

core/database.py
Реализовать подключение к PostgreSQL с использованием параметров из docker-compose.
Предоставить объект сессии/подключения для использования в CRUD-операциях.

Функциональный контур взаимодействия:
Фронтенд собирает лид-метрики и техническую информацию о поведении пользователя.
FE формирует единый пакет данных и отправляет в backend.
Backend записывает данные в таблицы Lead и Behavior (связь 1-к-1).
Данные хранятся в серверной БД локально на той же машине.
